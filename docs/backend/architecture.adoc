== Структура компонентов
Бэкенд системы «Умная теплица» реализован как REST API и выступает промежуточным слоем между веб-интерфейсом, IoT-платформой ThingsBoard и базой данных.

=== Контроллеры (входная точка)
Компоненты, обрабатывающие HTTP-запросы от веб-приложения:

- **Auth Controller** — управляет аутентификацией (`POST /login`, `POST /register`).
- **Device Controller** — отвечает за привязку и управление устройствами (`POST /devices/add`).
- **Telemetry Controller** — предоставляет исторические и текущие данные с датчиков (`GET /telemetry/{deviceId}`).
- **Notification Controller** — управляет настройками и историей уведомлений.

=== Сервисы (бизнес-логика)
Содержат основную логику приложения, независимо от протоколов:

- **User / Device / Telemetry / Notification Service** — реализуют правила валидации, преобразования данных, проверки прав доступа.

=== Инфраструктурные адаптеры
Обеспечивают взаимодействие с внешними системами:

- **ThingsBoard Gateway** — инкапсулирует вызовы REST API ThingsBoard (получение телеметрии, отправка команд).
- **Database Repository** — единый интерфейс для работы с PostgreSQL (паттерн Repository).
- **External Notification Clients** — клиенты для отправки уведомлений (Email через SMTP/SendGrid, сообщения через Telegram Bot API).

== Внешние зависимости

- **ThingsBoard IoT** — облачная IoT-платформа для приёма данных от ESP32-устройств по MQTT и хранения телеметрии.
- **PostgreSQL** — реляционная СУБД для хранения:
  - учётных записей пользователей,
  - метаданных устройств (Device ID, привязка к пользователю),
  - истории событий и настроек уведомлений.
- **Сервисы уведомлений** — внешние API (Telegram, Email) для доставки оповещений.

== Принципы проектирования

- **Разделение ответственности**: контроллеры не содержат бизнес-логики.
- **Инверсия зависимостей**: сервисы зависят от абстракций (интерфейсов), а не от конкретных реализаций.
- **Инкапсуляция внешних систем**: замена ThingsBoard или СУБД требует изменения только соответствующего адаптера.
- **Масштабируемость**: каждый компонент может быть вынесен в отдельный микросервис при росте нагрузки.

== Поток данных (пример)
1. Пользователь нажимает «Добавить устройство» во фронтенде.
2. Запрос приходит в **Device Controller**.
3. **Device Service** валидирует Device ID и Access Token.
4. **ThingsBoard Gateway** проверяет существование устройства в ThingsBoard.
5. При успехе — данные сохраняются через **Database Repository**.
6. Пользователь получает подтверждение.
